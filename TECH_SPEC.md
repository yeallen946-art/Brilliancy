## 1. Architecture Overview

**Local-first iOS app + offline content pipeline + static CDN. No application server in V1.**

```
┌─────────────────────────────┐      ┌──────────────────────────────┐
│  Content Pipeline (offline)  │      │  iOS App (SwiftUI)           │
│  runs on dev Mac, by Claude  │      │                              │
│  Code                        │      │  Bundled content.sqlite      │
│                              │      │  + downloaded packs          │
│  PGN sources                 │      │                              │
│   → curation                 │ ───▶ │  Daily challenge: fetches    │
│   → Stockfish analysis       │ CDN  │  static JSON from CDN        │
│   → LLM annotation           │      │                              │
│   → engine validation        │      │  StoreKit 2 (sub + lifetime) │
│   → human review             │      │  Local stats (SQLite/GRDB)   │
│   → pack build (sqlite/json) │      │  iCloud KV sync (streak,     │
└─────────────────────────────┘      │  rating) — optional V1.1     │
                                      └──────────────────────────────┘
```

Rationale: no backend = no ops cost, no auth, no privacy surface. Daily challenge is a pre-published static JSON per date on a CDN (Cloudflare Pages/R2 or GitHub Pages). The whole year of daily challenges can be generated in advance and published on a schedule.

## 2. Stack

| Layer | Choice | Notes |
|---|---|---|
| UI | SwiftUI, iOS 17+ | Observation framework, no UIKit unless necessary |
| Chess rules | `ChessKit` (Swift) or custom | Need: legal move gen, FEN/PGN, SAN. Evaluate ChessKit first; write custom only if it falls short |
| Persistence | GRDB (SQLite) | Content DB (read-only, bundled) + user DB (guesses, rating history) |
| IAP | StoreKit 2 | Subscription group (monthly/annual) + non-consumable lifetime |
| Analytics | TelemetryDeck | Privacy-friendly, no ATT prompt needed |
| Crash | Apple MetricKit / Xcode Organizer | No Firebase |
| CDN | Cloudflare Pages (static) | `/daily/2026-06-07.json`, content pack downloads |
| Pipeline | Python 3.12 + python-chess + Stockfish 16 + Anthropic API | Lives in `pipeline/` in same repo |
| Project gen | **XcodeGen** (`project.yml`) | The `.xcodeproj` is generated, not committed (see §2.1) |

Dependency policy: minimize. Each new SPM package needs justification in PR description.

### 2.1 Project generation (XcodeGen) — critical for the cross-machine workflow

The Xcode project is **defined declaratively in `project.yml`** and generated by XcodeGen; the `.xcodeproj` is git-ignored, not committed. Rationale: development happens on Windows (Claude Code, no Xcode) and builds on macOS. Hand-maintaining `.pbxproj` across machines is the single worst pain point of this setup — XcodeGen removes it.

- XcodeGen **globs source files by folder**, so adding a file = create the `.swift` in the right directory. No target membership editing, no `.pbxproj` edits, no merge conflicts.
- `project.yml` is plain text — Claude Code can author and edit it from Windows safely.
- macOS side regenerates with one command before building (see CLAUDE.md / AGENTS.md commands).
- `.gitignore` must include `*.xcodeproj/` (and `.xcodegen` cache); `project.yml` IS committed.
- Anything that can't be expressed in `project.yml` (rare: certain capabilities/entitlements) is configured in `project.yml`'s `settings`/`entitlements`, not by hand-editing the generated project.

### 3.1 GuessSession (core loop)

State machine per game:

```
LOAD → SHOW_CONTEXT → [for each guess point:]
  PRESENT_POSITION → AWAIT_GUESS → SCORE → REVEAL(master move + annotation)
  → AUTOPLAY(opponent reply + skipped moves) → next point
→ GAME_SUMMARY(total score, rating delta, share/upsell)
```

- Guess points are precomputed by the pipeline (`is_guess_point` flag per ply).
- Opening moves and forced sequences autoplay at ~0.5s/move with board animation.
- User plays the master's color for the entire game.

### 3.2 Scoring

Per guess point, the content DB stores: master move, engine eval of position, eval of top-5 moves (cp, from White's perspective, normalized to mover's perspective at runtime).

```
score(guess):
  if guess == master_move:            100  # "Match"
  elif eval_delta(guess) <= 30cp
       and guess in engine_top5:       85  # "Equally strong"
  elif eval_delta(guess) <= 80cp:      55  # "Reasonable"
  elif eval_delta(guess) <= 150cp:     25  # "Inaccuracy"
  else:                                 0  # "Blunder" (flag red)
eval_delta = eval(best_move) - eval(guess), clamped at mate scores
```

Game score = mean of point scores. Tune constants during TestFlight; keep them in one config struct.

### 3.3 Guess Rating (Elo-like)

- User rating `R` starts at 1200. Each guess point has a difficulty `D` (precomputed: based on % of engine-top-move obviousness — eval gap between best and second-best move, plus move type).
- Update: `R' = R + K * (s - E)` where `s` = normalized point score (0..1), `E = 1 / (1 + 10^((D - R)/400))`, `K = 16` (24 for first 100 guesses — fast early movement, per PRD §4.3 the curve should feel generous early).
- Store every rating snapshot for the history chart.
- Weakness breakdown: each guess point is tagged (`tactical | positional | endgame | opening | defense`) by the pipeline; aggregate hit-rate per tag.

## 4. Data Model

### content.sqlite (read-only, built by pipeline)

```sql
games(id, white, black, event, year, result, eco, hero_color,
      title,            -- "Fischer's Game of the Century"
      narrative_intro,  -- 2-3 sentence context, LLM-written
      pack_id, ply_count)
moves(game_id, ply, san, uci, fen_before,
      is_guess_point, difficulty, tags,        -- tags: JSON array
      eval_cp, eval_mate,                      -- position eval before move
      top5,                                    -- JSON: [{uci, cp, mate}]
      annotation,                              -- why the master move works
      wrong_move_notes)                        -- JSON: {uci: "why this fails"} for 2-3 common wrong guesses
packs(id, name, kind,          -- player | theme | daily_archive
      description, price_tier, -- free | premium
      sort_order)
```

### user.sqlite (GRDB, local)

```sql
guesses(id, game_id, ply, guessed_uci, score, created_at)
game_results(game_id, total_score, completed_at)
rating_history(rating, created_at)
streak(current, longest, last_completed_date)   -- single row
```

### Daily challenge JSON (CDN)

`https://cdn.<domain>/daily/{YYYY-MM-DD}.json` — one full game in the same shape as content DB rows, plus `daily_id`. Published ahead of time; app caches today + tomorrow.

## 5. Content Pipeline (`pipeline/`)

Python CLI, stages are idempotent and individually runnable:

```
pipeline/
  1_ingest.py     # parse PGN (sources: Lichess elite DB, public-domain
                  #   classic game collections), normalize, dedupe
  2_curate.py     # candidate scoring: famous players, decisive results,
                  #   sacrifice/tactic density; output curation list for
                  #   human pick (Jerry/son selects the 50 MVP games)
  3_analyze.py    # Stockfish: eval + top5 per position (depth ~22 or
                  #   200ms/pos); mark guess points: skip book moves
                  #   (first ~8 plies unless novelty), skip forced
                  #   recaptures, target 15-30 points/game; difficulty
                  #   + tag assignment
  4_annotate.py   # Claude API: generate narrative_intro, annotation,
                  #   wrong_move_notes. Prompt receives engine lines and
                  #   MUST ground every claim in them. Batch API for cost.
  5_validate.py   # automated checks: every move mentioned in annotation
                  #   is legal and appears in engine top5/PV; eval
                  #   adjectives match numbers ("winning" needs >+2.0);
                  #   length limits; reading level
  6_review.py     # emits review HTML (board + annotation side by side)
                  #   for human review; tracks approved/rejected per game
  7_build.py      # approved games → content.sqlite + daily JSON files
```

**Annotation grounding contract (critical, see PRD §6):** the LLM prompt includes engine PV and top-5 for the position; the system prompt forbids chess claims not derivable from the provided lines; `5_validate.py` rejects annotations referencing moves outside engine output. Rejected items go back to step 4 with the validation error in the retry prompt.

Cost note: ~50 games × ~25 points × (1 annotation + 2-3 wrong-move notes) ≈ manageable in Claude Max / Batch API budget; regenerate only failed items.

## 6. Monetization Implementation

- StoreKit 2, products: `sub.monthly` ($4.99), `sub.annual` ($29.99, 7-day trial), `lifetime` ($49.99 non-consumable).
- Entitlement check: `Transaction.currentEntitlements` on launch + listener. Single `EntitlementStore` observable, injected; **all gating goes through one `isPremium` flag** — no scattered checks.
- Free tier: daily challenge (today only), 3 marked sample games, scoring without annotations beyond first game (exact gating per PRD §7).
- Paywall triggers: post-daily-challenge upsell, locked pack tap, locked archive tap.
- Use SubscriptionStoreView where possible; App Store Small Business Program (15%).

## 7. Share Card

Render offscreen SwiftUI view → UIImage: app name, date, score band (emoji row per guess point: 🟩 match / 🟨 close / 🟥 miss — no move spoilers), streak. Share via `ShareLink`.

## 8. Analytics Events (TelemetryDeck)

`daily_started/completed`, `game_started/completed`, `guess_made(score_band)`, `paywall_shown(trigger)`, `purchase(product)`, `share_tapped`. No PII, no position data.

## 9. Testing

- **Unit (required):** Scoring + rating math (pure functions — table-driven tests), SAN/FEN round-trips, guess-point gating of pipeline output, entitlement gating logic.
- **Pipeline (required):** `5_validate.py` has its own test suite with known-bad annotations.
- **Snapshot:** BoardView rendering, share card.
- **Manual/TestFlight:** IAP flows (sandbox), full game UX with seed users.

## 10. Out of Scope (V1)

Backend/auth, Android, real-time engine analysis on device of arbitrary user moves beyond stored top-5 (everything precomputed; if guess not in top-5, score from stored eval via "not in top 5" bucket — see Scoring), localization, iPad-optimized layout (runs scaled), Game Center.

**Edge case to handle explicitly:** user guess not in stored top-5 → treat as `eval_delta > 80cp` minimum; never run engine on device in V1.
